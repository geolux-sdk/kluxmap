# AGENTS.md

## 기본 작업 원칙

- 답변과 진행 보고는 한국어로 작성한다.
- 작업을 시작하기 전에 `git status --short`로 현재 변경사항을 확인한다.
- 사용자가 만든 변경사항은 명시적인 요청 없이 되돌리지 않는다.
- 변경 범위는 요청된 작업에 필요한 파일과 동작으로 좁게 유지한다.
- 삭제 작업은 대상 목록을 먼저 확인하고, 프로젝트 내부 경로인지 검증한 뒤 진행한다.
- 임시 백업 파일이나 불필요한 백업 폴더는 남기지 않는다. 예: `backup/`, `*.bak`, `*.py_bak`.

## 수정 전 검토

- 코드, 설정, 문서 파일을 수정하기 전에 먼저 수정 대상 파일과 변경 내용을 사용자에게 보여준다.
- 사용자가 검토하고 승인한 뒤에만 실제 파일 수정이나 코드 작성을 진행한다.
- 사용자가 "먼저 검토", "수정하지 않고", "검토 받고"와 같이 말하면 절대 바로 수정하지 않는다.
- 승인 전에는 구현 대신 변경안, 예상 diff, 영향 범위, 검증 방법을 설명한다.
- 사용자가 이미 승인했거나 "그대로 적용", "진행", "수정해줘"라고 답한 뒤에만 파일을 변경한다.

## 검증 규칙

- Python 파일을 수정한 뒤에는 가능한 경우 `python -m py_compile <파일>`로 문법 검사를 수행한다.
- `settings.json`을 수정한 뒤에는 `python -m json.tool settings.json`으로 JSON 형식을 확인한다.
- 하드웨어 제어, ADC, relay, serial 관련 변경은 timing, retry, cleanup, READY/FAULT 상태 전환에 미치는 영향을 함께 검토한다.
- 검증 명령을 실행하지 못했으면 그 이유를 최종 답변에 명확히 남긴다.

## 커밋과 PUSH

- 커밋 전에는 `git diff --stat` 또는 필요한 diff를 확인해 의도한 변경만 포함됐는지 점검한다.
- 커밋 메시지는 작업 내용을 알 수 있게 작성한다. 삭제, 정리, 동작 변경, 검증 결과가 있으면 본문에 남긴다.
- `PUSH` 요청이 있어도 바로 푸시하지 않는다.
- PUSH 전에는 커밋 해시, 브랜치, 원격 저장소, 푸시될 변경 요약을 사용자에게 다시 보여주고 반드시 확인을 받는다.
- 사용자의 최종 확인 후에만 `git push`를 실행한다.

## 응답 방식

- 파일 위치나 코드 위치를 설명할 때는 가능한 한 정확한 파일 경로와 줄 번호를 제시한다.
- 로그나 오류 분석은 실제 로그와 코드 흐름을 기준으로 시간 순서대로 설명한다.
- 추측이 필요한 경우 추측임을 분명히 밝히고, 확인 가능한 방법을 함께 제시한다.
